# Progress story — Surface E hero (Phase 4)

**Status:** Phase 4 / B3.1 — shipped 2026-04-27
**Authority:** D-2026-04-27-12 (adaptive TDEE always-on, confidence
is metadata) + D-2026-04-27-17 (Progress is a weekly story not a
stat-card dashboard)
**Spec:** `docs/specs/2026-04-27-production-design-spec.md` Surface E

## What changed

Progress used to lead with a stat-card dashboard. Phase 4 replaces
the visual lead with an engine-generated narrative: the Adaptive TDEE
recap line is now the headline, with body copy that explains *why*
maintenance moved or stayed put, and a `<ConfidenceChip>` inline.

The existing maintenance card / weight chart / calorie chart remain
on the page beneath the hero — they're demoted to the supporting
trend layer, not the visual lead.

## Component map

- **Engine commentary lib (shared):** `src/lib/nutrition/progressCommentary.ts`
  — `generateProgressCommentary`, `splitBodyIntoSegments`. Pure
  function, no I/O. Mobile re-export at `apps/mobile/lib/progressCommentary.ts`.
- **Web primitive:** `src/app/components/suppr/progress-headline.tsx`
  — accepts the resolved `ProgressCommentaryResult` and renders the
  eyebrow / headline / body / inline ConfidenceChip.
- **Mobile primitive:** `apps/mobile/components/today/ProgressHeadline.tsx`
  — RN mirror of the web component.
- **Web host:** `src/app/components/ProgressDashboard.tsx` —
  inserts the `<ProgressHeadline>` between `<HouseholdBar>` and the
  range-picker pills.
- **Mobile host:** `apps/mobile/app/(tabs)/progress.tsx` — same
  insertion point.

## Three regimes

The commentary lib decides which voice to use:

| Regime         | Trigger                                              | Headline example                                  |
| -------------- | ---------------------------------------------------- | ------------------------------------------------- |
| `adjustment`   | `abs(currentTdee - prevWeekTdee) > 30 kcal`           | "Your maintenance adjusted up by 60 kcal"         |
| `calibrating`  | `current=null` OR `confidence='low'` OR `loggingDays<14` | "Welcome — we'll start estimating maintenance after your first week" / "We're still calibrating your maintenance" |
| `steady`       | confidence ≥ medium AND `delta ≤ 30 kcal`             | "Maintenance held steady this week"               |

The chip level is taken straight from the engine's confidence — it's
displayed as metadata, never as a gate. D-2026-04-27-12 binds.

## Voice rules (pinned by tests)

- UK English (`behaviour`, `personalise`, `kilocalories`, `fibre`).
- Second-person — `your`, `you`, `we`. Never `the user`.
- No exclamation marks.
- Numerics rendered with `tabular-nums` font-variant.

`tests/unit/progressCommentaryPhase4.test.ts` pins all three regimes
+ the voice rules. `progressHeadlinePhase4.test.tsx` (web) +
`apps/mobile/tests/unit/progressHeadlinePhase4.test.tsx` (mobile)
pin the render shape.

## State coverage (per spec)

- **Default**: real engine data → `steady` or `adjustment`.
- **Loading**: caller renders the skeleton then swaps in the
  `<ProgressHeadline>` once `current` is non-null. Hero never shows
  "loading" copy itself.
- **Empty (first 3 days)**: `calibrating` regime + welcome copy.
- **Partial (mid-warmup, 3 ≤ days < 14)**: `calibrating` regime +
  "Need a few more days of data" copy. Trend cards still render.
- **Offline**: cached commentary renders. The host handles the offline
  banner separately (existing behaviour, not new).
- **Confidence is `low`**: still renders — confidence chip says
  `Low confidence`, body adapts, value is suppressed if it's
  speculative.

## Deferred — prev-week + loss-week telemetry

The `ProgressCommentaryInput` accepts `prevWeekTdee` and
`avgIntakeOnLossWeeksKcal` so the `adjustment` regime can fire and
quote the "average intake on loss weeks" line. The host wiring
currently passes neither (the weekly history isn't yet persisted).
This is intentional; the commentary collapses to `steady` /
`calibrating` until we add a `progress_weekly_tdee_history` migration
and a "loss-week" aggregate. See Phase 5 backlog.

## Cross-platform parity

- Same primitive copy on both platforms (eyebrow `THIS WEEK`,
  headline strings, body strings).
- Same regime decision logic (one shared `progressCommentary.ts`).
- Same `<ConfidenceChip>` placement (inline at the end of the body).
- Layout: web SupprCard 16pt radius, padding 5; mobile SupprCard
  16pt radius, padding 20×16. Both use `--elev-card` /
  `Elevation.card`.
- No intentional divergence. `sync-enforcer` carve-out: none.
