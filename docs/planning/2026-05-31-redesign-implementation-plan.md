### [P2] Add hardcoded-hex / raw-colour lint guard to prevent token drift returning — NEW
- owner: design-system-enforcer · launch-blocker: False · depends: Activate Elevation.card token (ENG-795); Enforce one commit-action colour (ENG-796); Unify brand mark (ENG-797)

**Problem.** The three P0 token changes (elevation, commit colour, brand mark) correct today's drift. Without a mechanical guard, the same drift will return: a developer spreads a raw `#588CE4` hex, `rgba(88,140,228,0.10)`, or `text-violet-600` Tailwind utility instead of the semantic token. Currently there is no ESLint rule or CI check preventing this. The existing `--max-warnings 500` ratchet catches lint warnings in aggregate but there is no specific no-hardcoded-colour rule.

**Goal.** A CI-enforced guard that fails (warn → eventually error) when raw hex colours or non-token Tailwind colour utilities appear in component source files, making the token system self-defending going forward.

**Acceptance criteria.**
- A `scripts/check-design-tokens.mjs` script that:
  - Greps `apps/mobile/**/*.{ts,tsx}` for raw hex literals (`/#[0-9a-fA-F]{3,6}/`) that are NOT inside `constants/theme.ts` (which IS the token definition file — exempted).
  - Greps `src/**/*.{ts,tsx,css}` for `text-violet-*`, `from-violet-*`, `to-violet-*`, `to-indigo-*` Tailwind utilities (the specific class of web drift found in this audit).
  - Greps both trees for hardcoded `rgba(` calls outside token definition files.
  - Exits non-zero and prints offending file + line if any match found.
- Script added to `npm run ci` chain (after typecheck, before build) OR as a named `npm run check:tokens` script called from CI yaml.
- Exemptions file (comment in script or a `.tokenignore` list) for legitimate exceptions (e.g. `colors.overlay = '#00000088'` in theme.ts is the definition, not a violation; any `#000` in SVG `fill` props is a known exemption).
- `tests/unit/checkDesignTokens.test.ts` runs the script against a fixture directory containing one known violation and asserts non-zero exit.
- No feature flag required.
- Parity: script covers both platforms in one pass.

**Dependencies.** Should run AFTER ENG-795/796/797 land so the first run starts clean (otherwise it would fire on the current violations). Sequence: provision flags → token fixes → add guard.

**Files.** `scripts/check-design-tokens.mjs` (new), `.github/workflows/ci.yml` (add script step), `tests/unit/checkDesignTokens.test.ts` (new).

**Feature flag:** gate spring vocabulary and morph behind `isFeatureEnabled('design-system-motion-v1')` on both platforms. Old paths (plain `withTiming` / no morph) survive in the `else`.

**Parity:** `src/lib/preferences/premiumMotion.ts` is already marked as `@suppr/shared` so web and mobile share the constants. The web CSS ease strings live alongside them.

**Acceptance criteria:**

### Dev / E2E flag-override convention (`EXPO_PUBLIC_FLAG_FORCE_*`)

PostHog's local-evaluation cache races first render in dev, so a Maestro flow
(or a local sim check) can't reliably hit a flag-on branch via PostHog
targeting alone. Both `isFeatureEnabled` and `isFeatureDisabled` in
`apps/mobile/lib/analytics.ts` therefore honour an env-var override **only when
`__DEV__` is true** — production builds drop the whole branch via Hermes DCE, so
the override is inert in shipped apps.

- **Env-var name:** `EXPO_PUBLIC_FLAG_FORCE_<FLAG_KEY>`.
- **Mapping rule:** uppercase the flag key and **replace hyphens with
  underscores** (env-var names can't contain hyphens). So
  `redesign_search_results` → `EXPO_PUBLIC_FLAG_FORCE_REDESIGN_SEARCH_RESULTS`,
  and a hyphenated working key like `weekly-checkin-card` →
  `EXPO_PUBLIC_FLAG_FORCE_WEEKLY_CHECKIN_CARD`.
- **Values:** `"true"` forces the flag ON, `"false"` forces it OFF. Any other
  value (or unset) falls through to the live PostHog client.
- **`EXPO_PUBLIC_` prefix** is required so Expo inlines the var into the client
  bundle at build time.

Example — force the win-moment path on for a Maestro capture:

```bash
EXPO_PUBLIC_FLAG_FORCE_REDESIGN_WINMOMENT=true npm run ios:simulator
```

The web mirror (`src/lib/analytics/track.ts`) honours the same contract under a
`NEXT_PUBLIC_` prefix (added 2026-05-31, ENG P5 gap #13): `isFeatureEnabled` /
`isFeatureDisabled` read `NEXT_PUBLIC_FLAG_FORCE_<FLAG_KEY>` whenever
`NODE_ENV !== "production"`, accepting `"1"`/`"true"` (force ON) or
`"0"`/`"false"` (force OFF) — same uppercase + hyphen→underscore mapping as
mobile. This lets Playwright visual specs capture flag-ON goldens (the committed
auth fixture seeds an empty PostHog flag set, so without a force layer only the
flag-OFF path renders). The override is inert in production. Example:

```bash
NEXT_PUBLIC_FLAG_FORCE_REDESIGN_BRANDED_SHEETS=true npm run dev
```

Keep the flag **key** identical across platforms; only the env-var prefix differs
(`NEXT_PUBLIC_` on web, `EXPO_PUBLIC_` on mobile).

## Colour lint guard (ENG-811)

To stop the token drift this redesign corrects from quietly returning, a
`no-restricted-syntax` colour guard runs at lint time on the web tree
(`eslint.config.mjs`, `SUPPR_RAW_COLOUR_SYNTAX`). It flags:

1. **Raw hex colour string literals** — `"#588CE4"`, `'#fff'`, `"#ff00aa80"`
   (anchored, so anchor hrefs like `#section` are never matched).
2. **Raw Tailwind colour utilities** — `text-violet-600`, `bg-blue-500`,
   `from-indigo-600`, etc. — in both plain string literals and template-literal
   `className` strings.

**Allowlist (where literal colour values are correct):**
- `src/styles/theme.css` — the web token definitions. CSS, so naturally outside
  the rule's `src/**/*.{ts,tsx}` glob; never linted.
- `apps/mobile/constants/theme.ts` — the mobile token definitions. Lives in the
  mobile tree (guarded by `apps/mobile/eslint.config.js`, not the web config).
- `tests/**` — fixtures legitimately assert literal colour values; the rule is
  scoped to `src/**` + `app/**` only and excludes tests.

**Severity: `warn` (intentional).** The web tree carries ~400 pre-existing raw
hex + raw-Tailwind-colour usages (a mix of real drift to migrate and legit
exceptions — e.g. third-party social brand hexes in `source-badge.tsx`).
Shipping at `error` would red CI immediately. At `warn` the drift is visible in
lint output and stays bounded by the existing `--max-warnings 500` cap
(currently ~394 total warnings, comfortably under).

**How to flip to `error` later:** once the redesign token sweep migrates the
existing usages down to the allowlist and `npm run lint` reports zero of these
two selectors firing, change `"warn"` → `"error"` on the
`no-restricted-syntax` rule entry in `eslint.config.mjs` **and** in the mobile
counterpart (`apps/mobile/eslint.config.js`, ENG-811 mobile lane). Verify a zero
count first, or CI reds.
