### F-10 — infinite scroll + drop images in food-search rows

- **Cause (pagination):** Both `FoodSearchModal.tsx` (mobile) and `FoodSearch.tsx` (web) fetched a single page of results from each of USDA FDC, OpenFoodFacts, and Edamam, then merged + sliced to ~24–30 rows. USDA's route forwarded `pageSize: 10` without a `pageNumber` so the server itself also returned only page 1. Users ran out of rows the moment they scrolled past the initial slice; there was no `onEndReached` / IntersectionObserver path to request more.
- **Cause (image inconsistency):** The build 10 A2 per-serving work introduced product images on rows that had `imageUrl` (Edamam restaurant + USDA Branded hits); USDA generic hits had no image. The mixed rows looked visually uneven.
- **Fix — pagination (primary paginator = USDA):**
  - `src/lib/usda/fdcClient.ts` — `fdcFoodsSearch` accepts `{ pageNumber?: number; pageSize?: number }` and sets both in the POST body (1-indexed per USDA spec).
  - `app/api/usda/search/route.ts` — reads `?page=<n>`, parses to a safe positive integer, forwards to the client, echoes `page` on the envelope.
  - `app/api/edamam/search/route.ts` — Edamam's `parser` endpoint is not natively paginated, so `page > 1` short-circuits with `{ ok: true, mode, page, hits: [] }`. USDA carries the load past page 1; client-side de-dup handles the merge cleanly.
  - `apps/mobile/lib/verifyRecipe.ts` — `searchUsda`, `searchOpenFoodFacts`, `searchEdamam`, `searchFoods`, and `mergeResults` all take an `opts.page` (1-indexed). OFF uses its native `&page=` parameter. `searchFoods` also takes an `opts.limit` (default 24).
  - `apps/mobile/components/FoodSearchModal.tsx` — declares `pageRef` + `hasMoreRef` + `loadingMore` state. `useEffect(visible)` and `onChangeText` both reset both refs on every new query. `loadMore` is invoked by `FlatList#onEndReached` (`onEndReachedThreshold={0.4}`); uses a new `appendPage` helper to drop rows whose key is already present; flips the terminal latch when either the page is empty or fully duplicated.
  - `src/app/components/FoodSearch.tsx` — same state shape. Trigger is an `IntersectionObserver` watching a hidden `sentinelRef` `<div>`; `rootMargin: "120px 0px"` starts the fetch slightly before the true bottom. Web's `appendPage` de-dupes by BOTH row key AND normalised name so cross-source overlap is caught even when the key differs.
  - Custom-food rows are excluded from the paginated fetch entirely — page-invariant and always on top.
- **Fix — image consistency (drop entirely on search rows):**
  - `apps/mobile/components/FoodSearchModal.tsx` — dropped the `Image` import from `react-native`, the `{item.imageUrl && <Image … />}` JSX, and the `productImage` StyleSheet entry. The `imageUrl` field is left in place for future detail-screen use.
  - `src/app/components/FoodSearch.tsx` — never rendered an `<img>` on a search row. Structural pin forbids regressions.

## 2026-04-19 — F-3 + F-4: Maintenance unification + weight progress math + chart range filter

One commit, three related fixes — all three sit on the Progress screen and share state shapes.
- **Cause (F-3):** Today's Activity Bonus tile computed maintenance as static Mifflin × activity multiplier only (`apps/mobile/app/(tabs)/index.tsx:profileMaintenanceTdeeKcal` + `src/app/components/NutritionTracker.tsx:profileMaintenanceTdee`). Progress's "Your TDEE" card read adaptive TDEE via `getEffectiveTDEE` whenever confidence ≥ medium. Same user, same day, two numbers, two labels — the tester's 1,675 / 1,777 screenshot was entirely predictable. On top of that the two label strings ("Maintenance" vs "Your TDEE") taught users to think of them as different concepts.
- **Cause (F-4a):** Mobile `apps/mobile/app/(tabs)/progress.tsx` had a `Math.max(3, Math.min(100, Math.round(journeyProg.pct * 100)))` floor on the progress bar's percentage so a genuine 0% rendered as 3% and the "(3% of the way)" copy looked like a broken tracker. The web version (`ProgressDashboard.tsx`) had the same `Math.max(progressPct, 3)` floor on the bar's width. Day-one users saw "3% progress" before they'd moved a kilogram.
- **Cause (F-4b):** The range-button filter on `apps/mobile/app/weight-tracker.tsx` lived inline as a private `filterByRange(map, range)` helper. It worked correctly in code — `>= cutoffStr` filter, `daysForRange` mapped each range to the right number of days — but because it was inlined there was no unit-level pin for per-range behaviour, and the tester's screenshot (chart looked identical across 3M/6M/9M) wasn't ever proven at the filter boundary. In practice the bug is most often "user has <90 days of data so 3M/6M/9M all return the same set" which the chart can't distinguish from a broken filter.
- **Fix — shared resolver:**
  - New `src/lib/nutrition/resolveMaintenance.ts` exports `resolveMaintenance(profile, { now? }) → { kcal, source: "adaptive" | "formula", confidence, formulaKcal, adaptiveRejectedAsStale } | null`. Adaptive wins when `adaptive_tdee_confidence` ∈ {medium, high} AND `adaptive_tdee > 0` AND the value is not stale (> `ADAPTIVE_STALE_DAYS` = 14 days since `adaptive_tdee_updated_at`). Stale adaptive is rejected silently → falls back to formula, with `adaptiveRejectedAsStale: true` on the return so a future "re-weigh to refresh" hint has the signal it needs. Missing `updated_at` does NOT mark the value stale (pre-2026-04-14 rows shipped without the column). Formula = Mifflin × activity multiplier via `calculateTDEE`. Returns `null` when formula inputs are incomplete (never fabricates a kcal number).
  - Also exports `buildMaintenancePopoverCopy(resolved) → string` — canonical info-popover copy: `"Maintenance is the calories you'd burn in a normal day. Based on your actual intake and weight changes (${confidence} confidence)."` when adaptive, `"…Formula estimate from your stats and activity level."` when formula.
  - `ADAPTIVE_STALE_DAYS` is exported so tests can reference the exact cutoff.
- **Fix — Progress card (F-3 §3):** both `src/app/components/ProgressDashboard.tsx` and `apps/mobile/app/(tabs)/progress.tsx` renamed the card title from "Your TDEE" → "Maintenance" and swapped the big-number source from `isAdaptive ? adaptiveTdee : staticTdee` to `resolveMaintenance(...).kcal`. The ADAPTIVE badge, confidence bars, "Formula estimate: {n} kcal ({+N actual})" subline, and data-progress bars are all preserved — power users still see the spread; the label unification just stops the tester's confusion. The load path now selects `adaptive_tdee_updated_at` alongside the existing adaptive columns so the stale-check has real data.
- **Fix — Today tile (F-3 §4):** `src/app/components/NutritionTracker.tsx` (web) and `apps/mobile/app/(tabs)/index.tsx` (mobile) both call `resolveMaintenance(...)` with the same profile shape. The web select gained `adaptive_tdee_confidence, adaptive_tdee_updated_at`; the mobile select gained `adaptive_tdee, adaptive_tdee_confidence, adaptive_tdee_updated_at`. Both hosts now forward an extra `maintenanceSource` + `maintenanceConfidence` prop to the TodayActivityBonusCard so the info popover renders the canonical shared copy. The card keeps its existing BMR × multiplier breakdown as a fallback for hosts that haven't wired the source yet — no regression for pre-ship callers.
- **Fix — F-4a weight progress math:** new shared helpers in `src/lib/weightProjection.ts`:
  - `computeWeightJourneyProgressPct({ startKg, currentKg, goalKg }) → number | null` — canonical formula `(start - current) / (start - goal)` clamped to `[0, 1]`. Direction-aware (lose journeys have span > 0, gain journeys span < 0; same expression handles both). Returns `null` when `|start - goal| < 0.1` (no meaningful journey to measure — avoids the divide-by-near-zero blow-up that produced tiny fake percentages). Returns `null` for non-finite inputs.

- **Fix — F-4b chart range filter:** new `filterByDateRangeDays(map, maxDays, now?)` helper in `src/lib/weightProjection.ts`. The weight-tracker's private `filterByRange(map, range)` now delegates to it (`filterByDateRangeDays(map, daysForRange(range))`) so the per-range behaviour is testable at the filter level. No behaviour change for users with ≥90 days of data; for users with thin history we make the failure mode visible at the test boundary.

## 2026-04-19 — F-13: Auto-track caffeine + alcohol from food logs

- **In-session product request (2026-04-19, gracehowse):** "caffeine and alcohol should be auto tracked when a user logs those sorts of items. For example if I log 2 glasses of wine and a coffee both should auto update." Previously the caffeine / alcohol daily totals in `HydrationStimulantsCard` were only fed by the card's own quick-add chips — a USDA / OFF / Edamam food log of "1 espresso" bumped macros but not the stimulant totals, so the user had to double-log (tap the chip + log the coffee).

- **Audit — which sources expose caffeine + alcohol per 100 g:**

  | Source | Caffeine | Alcohol | Notes |

  | USDA FDC (Foundation / SR Legacy / Survey / Branded) | nutrient `262` in `mg` | nutrient `221` (`Alcohol, ethyl`) in `g` | Inline-available on search + full-food; `fdcFoodMacrosPer100g` now surfaces both. |
