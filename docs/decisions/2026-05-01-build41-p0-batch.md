## P0-2 — Calorie ring solid green at-or-above target

- TestFlight ID: `AEvjNTAVsipFKDysDkJD2g4`
- Tester comment: "Why is the ring now gradient even when the user
  has logged instead of green?"

Build 40 used the indigo→pink brand gradient across the entire
consumed-vs-target range (post-`59cc821`). The user lost the "you're
done" solid-success signal once they hit target.

**Decision:** keep the gradient for the in-progress arc
(`consumed < goal`), switch to solid `Accent.success` (mobile) /
`var(--success)` (web) once `consumed >= goal`. Drop the
destructive-red-on-over branch — going over a daily calorie target is
normal tracking, not an error state. The centre text colour still
flips to warning amber on over (in `remaining` displayMode) so the
"how far over" signal stays unambiguous.

Files: `apps/mobile/components/charts/CalorieRing.tsx`,
`src/app/components/suppr/daily-ring.tsx`. Pinned by
`tests/unit/calorieRingSolidGreenAtTarget.test.ts`.

## P0-3 — Tracking extras quick-add chips persist

- TestFlight ID: `AEsaeOW2Qw-BQa29teBp-Ns`
- Tester comment: "Adding alcohol or coffee still not impacting these
  numbers"

The Build 40 round-3 fix (`eb5f36ce`, 2026-04-27) added
error/rollback to the caffeine + alcohol persist paths but kept the
`let persisted = null; setState((prev) => { persisted = next; return
next; });` pattern. React 18 invokes functional updaters lazily during
the next commit, so `persisted` was still `null` when the persist
branch on the next line checked it. The supabase write never fired,
the round-3 error path never ran, and the round-3 toast never
surfaced. On next focus the local state hydrated from the
(still-zero) server row → "reset to 0" symptom.

**Decision:** drop the closure-capture pattern across all four
quick-add handlers (`addCaffeineMg`, `addAlcoholG`, `addWaterMl`,
`resetHydrationStimulantsForDay`). Compute `next` synchronously from
the closure-captured map, call `setState(next)` directly (not
functional updater), persist with the directly-captured `next`, and
roll back to the captured `prev` on error.

Web's equivalent (`addCaffeineMgForSelectedDay` etc. in
`src/context/AppDataContext.tsx`) was always correct — it persists
INSIDE the updater closure, so the closure-capture trap never applied.
No web change required.

Files: `apps/mobile/app/(tabs)/index.tsx`. Pinned by
`apps/mobile/tests/unit/trackingExtrasPersist.test.ts`.

## P0-4 — Recipe Log honours `recipe.meal_type` first

- TestFlight ID: `AB1PYpfPjbd9li7jtnlAsIE`
- Tester comment: "Doesn't give me an option of which meal to log
  this for and it ended up logging it as lunch. Also this was a
  breakfast recipe and I marked it as such when I imported it."

The mobile recipe Log button delegated to a private
`journalSlotFromMealTypes` helper that hard-fell-back to "Lunch" when
`recipe.meal_type` was null or unmatched. Explicitly-tagged recipes
already worked, but a recipe imported without a tag and logged at 7pm
landed in Lunch — and there was no UI to override.

**Decision (Build 41 minimum):** extract the helper to
`src/lib/nutrition/recipeJournalSlot.ts` so it can be unit-tested and
shared with web's CookMode log path. Add a
`fallbackSlotFromTimeOfDay` ladder
(`Breakfast < 11 < Lunch < 15 < Snacks < 17 < Dinner`) and use it as
the fallback when meal_type is empty or unmatched. Web's CookMode
already had a near-identical ladder inline; it now imports the
shared helper for cross-platform parity.

**Decision (deferred):** an explicit meal-slot picker
(Breakfast / Lunch / Dinner / Snack bottom-sheet, with the inferred
slot pre-selected) is the better long-term answer per the user's
request. Out of scope for Build 41 — tracked separately so the
feature can land in a focused PR with proper UI design.

Files: `src/lib/nutrition/recipeJournalSlot.ts` (new),
`apps/mobile/app/recipe/[id].tsx`,
`src/app/components/CookMode.tsx`. Pinned by
`tests/unit/recipeJournalSlot.test.ts`.
