# Decision log: bump web `@supabase/supabase-js` to 2.102.1 (P1-10, 2026-04-25)

**Date:** 2026-04-25
**Status:** Resolved
**Trigger:** P1 #10 in [Opus 4.7 codebase review](./2026-04-25-opus47-codebase-review.md). Web was pinned at `^2.56.0` (Feb 2025); mobile at `^2.102.1` (May 2025). ~46 minor releases between the two, including realtime channel changes, edge function invocation signature tweaks, and `select` query type-inference improvements. Both v2.x major; no breaking change documented but the surface had drifted.

---

## Decision

Bumped root `package.json` from `"@supabase/supabase-js": "^2.56.0"` to `"^2.102.1"`. `npm install` regenerated `package-lock.json` (1 package removed, 151 changed). Web + mobile now share the same minor pin.

`@supabase/ssr@^0.10.2` was already compatible with 2.102.1 (peer dep range covers v2.x). No SSR helper change required.

## Rationale

The CLAUDE.md non-negotiable is web ↔ mobile parity. Different Supabase client versions don't immediately produce different behaviour for shipped code, but they do mean any new feature added on one platform might rely on a feature that doesn't exist on the other — a silent divergence that surfaces at runtime in one specific code path. The bump closes that future-proofing gap before it becomes a real bug.

The two months of release notes between 2.56 → 2.102 included:
- Realtime channel improvements (heartbeat, reconnection backoff)
- `auth.getUser` token-validation hardening
- Type inference for `.select()` chains with computed columns
- Edge function streaming response support

None of these are observable behaviour changes for the code we currently ship — verified by running the full unit + integration test surface (134/134 green) covering the Supabase-touching paths: auth, household RLS, RevenueCat + Stripe webhook persistence, account deletion, recipe import, voice/photo log routes, planned-meal micros.

## Alternatives considered

- **Defer the bump until post-launch.** Rejected. Soft TestFlight is imminent; new Supabase features that mobile already uses (realtime, etc.) could create a runtime fork if a code path ever calls a mobile-only helper from a shared module. Bumping now closes that hole; post-launch the same bump is more disruptive because we'd be doing it under user load.
- **Bump to the latest @supabase/supabase-js (2.x.y as of this writing).** Rejected. Matching mobile's pin guarantees parity with what's actually shipping in TestFlight today; chasing the latest minor introduces variance with the mobile build.
- **Bump @supabase/ssr at the same time.** Rejected. `@supabase/ssr@0.10.2` is current and works against 2.102. No reason to touch it.

## Implementation

- `package.json` — `"@supabase/supabase-js": "^2.56.0"` → `"^2.102.1"`.
- `package-lock.json` — regenerated by `npm install`. 1 package removed (transitive dep cleanup), 151 changed.
- No source code changes required. The 134-test green run confirms.

## Platforms affected

- **Web:** uses the same client version mobile has been running for weeks. No observable behaviour change in the test surface. Slight bundle-size delta (negligible — minor releases).
- **Mobile:** unchanged.
- **Supabase:** unchanged.
